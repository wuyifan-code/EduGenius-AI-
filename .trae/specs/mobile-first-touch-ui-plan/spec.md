# MediMate 移动优先触控 UI 开发计划

## 为什么
当前项目的 UI 组件缺乏触控优化，缺少实际功能交互。用户需要一个每个 UI 都可以触控、有实际作用的移动优先体验，确保应用在移动设备上能够提供流畅、直观的用户体验。

## 什么变化
- 为所有可交互元素添加触控优化（按钮、链接、表单控件）
- 实现每个 UI 组件的实际功能（而非仅展示）
- 添加触控反馈（按下效果、加载状态）
- 优化移动端布局和导航
- 添加手势操作支持（滑动、拖拽）

## 影响
- 受影响的规范：患者端、陪诊师端、管理员端、登录、消息、通知、个人资料、设置
- 受影响的代码：components/ 目录下的所有组件，App.tsx 路由

## 新增需求
### 需求：全局触控优化
系统应确保所有可交互元素符合移动触控标准

#### 场景：触控区域大小
- **当** 用户在移动设备上查看应用时
- **则** 所有可点击元素的最小触控区域为 44x44 像素

#### 场景：触控反馈
- **当** 用户按下可交互元素时
- **则** 元素应提供视觉反馈（颜色变化、缩放、震动）

### 需求：患者端功能完整性
患者端 UI 组件应具备完整功能

#### 场景：下单陪诊服务
- **当** 患者填写陪诊需求并提交时
- **则** 系统应创建订单并显示订单详情

#### 场景：查看订单列表
- **当** 患者查看订单列表时
- **则** 应显示订单状态、陪诊师信息、订单时间

### 需求：陪诊师端功能完整性
陪诊师端 UI 组件应具备完整功能

#### 场景：接单管理
- **当** 陪诊师查看新订单时
- **则** 可以选择接单或拒绝

#### 场景：订单状态更新
- **当** 陪诊师完成服务时
- **则** 可以更新订单状态并提交评价

### 需求：消息功能
实时消息系统应正常工作

#### 场景：发送消息
- **当** 用户发送消息时
- **则** 消息应实时显示在聊天窗口

### 需求：通知功能
通知系统应正常工作

#### 场景：接收通知
- **当** 有新通知时
- **则** 通知图标应显示未读数量Badge

## 修改需求
### 需求：导航栏适配
现有导航栏应适配移动端

#### 修改内容
- 底部导航栏应固定在底部
- 导航图标应足够大（至少 24px）
- 当前选中项应有明显指示

## 移除需求
- 无
