# MediMate 完整商用系统规范

## 为什么
当前 MediMate 项目虽然拥有完整的 UI 界面，但存在严重的功能缺失问题：
1. **假加载问题** - 个人资料、通知等页面永远显示"加载中..."
2. **搜索功能无效** - 搜索框只有 UI，无法实际搜索
3. **下单流程断裂** - 无法完成从选择陪诊师到支付的完整流程
4. **支付系统缺失** - 微信支付、退款等功能未实际对接
5. **交互元素无效** - 很多按钮、链接点击后无任何反应

这些问题导致产品无法投入商用，需要一个完整的系统化改造。

## 什么变化

### 第一阶段：修复假加载与错误处理
- 修复所有组件的加载状态处理，添加超时机制和错误降级
- 实现前端 Mock 数据层，确保后端不可用时前端仍能正常展示
- 添加网络错误提示和重试机制

### 第二阶段：实现真实搜索功能
- 实现医院/科室搜索 API 和前端对接
- 实现陪诊师搜索和筛选功能
- 实现搜索结果页面和排序功能

### 第三阶段：完整下单流程
- 实现从选择陪诊师到确认订单的完整流程
- 实现订单创建 API 和状态管理
- 实现订单详情页的所有功能按钮

### 第四阶段：支付系统对接
- 实现微信支付二维码生成和轮询
- 实现支付状态回调处理
- 实现退款申请和审核流程

### 第五阶段：消息系统完善
- 实现 WebSocket 实时消息
- 实现消息列表和对话详情
- 实现新消息提醒和推送

### 第六阶段：通知系统完善
- 实现通知生成机制（订单状态变更、支付成功等）
- 实现通知点击跳转对应页面
- 实现通知偏好设置持久化

### 第七阶段：管理后台完善
- 实现用户管理（编辑、禁用、删除）
- 实现订单管理（查看、取消、退款审核）
- 实现数据统计和导出功能

### 第八阶段：系统优化与测试
- 实现前端性能优化（懒加载、缓存）
- 实现完整的错误监控和日志
- 实现端到端测试覆盖核心流程

## 影响
- 受影响的规范：用户认证、搜索系统、订单系统、支付系统、消息系统、通知系统、管理后台
- 受影响的代码：所有 components/、services/、server/src/modules/
- 数据库变更：可能需要新增表或字段

## 新增需求

### 需求：前端错误处理与降级
系统应在前端实现完善的错误处理机制

#### 场景：API 请求失败
- **当** 后端 API 请求失败或超时时
- **则** 显示友好的错误提示，提供重试按钮
- **并且** 自动切换到 Mock 数据模式，确保页面可用

#### 场景：加载超时
- **当** 数据加载超过 5 秒时
- **则** 自动取消加载状态，显示"加载失败"提示
- **并且** 提供手动刷新按钮

### 需求：真实搜索功能
系统应提供可工作的搜索功能

#### 场景：搜索医院
- **当** 用户在搜索框输入关键词时
- **则** 实时显示搜索建议
- **当** 用户确认搜索时
- **则** 跳转到搜索结果页，显示匹配的医院列表

#### 场景：筛选陪诊师
- **当** 用户设置筛选条件（价格、评分、性别等）时
- **则** 实时更新陪诊师列表
- **并且** 支持排序（按评分、按价格、按距离）

### 需求：完整下单流程
系统应支持从选择到支付的完整下单流程

#### 场景：选择陪诊师并下单
- **当** 患者浏览陪诊师列表并点击"预约"时
- **则** 显示陪诊师详情和可预约时间
- **当** 患者选择服务时间并确认时
- **则** 跳转到订单确认页面，显示费用明细
- **当** 患者确认订单时
- **则** 创建订单并跳转到支付页面

#### 场景：查看和管理订单
- **当** 患者查看订单列表时
- **则** 显示所有订单及状态
- **当** 患者点击订单时
- **则** 显示订单详情，包含陪诊师信息、服务状态、操作按钮
- **并且** 支持取消订单、申请退款、联系陪诊师

### 需求：支付系统
系统应支持完整的支付流程

#### 场景：微信支付
- **当** 患者选择微信支付时
- **则** 调用后端生成支付二维码
- **并且** 前端轮询支付状态
- **当** 支付成功时
- **则** 自动跳转到支付成功页面，更新订单状态

#### 场景：申请退款
- **当** 患者在订单详情页点击"申请退款"时
- **则** 显示退款原因表单
- **当** 患者提交申请时
- **则** 创建退款申请，订单状态变为"退款中"
- **并且** 通知管理员审核

### 需求：实时消息系统
系统应提供即时通讯功能

#### 场景：发起对话
- **当** 用户点击"私信"或"联系陪诊师"时
- **则** 创建或打开对话窗口
- **并且** 支持发送文字、图片消息

#### 场景：实时接收消息
- **当** 用户处于对话页面时
- **则** 通过 WebSocket 实时接收新消息
- **并且** 显示未读消息数徽章

### 需求：通知系统
系统应提供完整的通知功能

#### 场景：生成通知
- **当** 订单状态变更时（创建、支付成功、服务完成等）
- **则** 自动生成通知给相关用户
- **并且** 支持推送通知到浏览器

#### 场景：点击通知跳转
- **当** 用户点击通知时
- **则** 根据通知类型跳转到对应页面
- **并且** 自动标记为已读

### 需求：管理后台
系统应提供完整的管理功能

#### 场景：用户管理
- **当** 管理员查看用户列表时
- **则** 显示所有用户信息，支持搜索和筛选
- **并且** 支持编辑用户信息、禁用/启用账号

#### 场景：订单管理
- **当** 管理员查看订单列表时
- **则** 显示所有订单及状态，支持按状态筛选
- **并且** 支持查看详情、取消订单、处理退款

#### 场景：退款审核
- **当** 管理员进入退款管理时
- **则** 显示待审核的退款申请列表
- **当** 管理员审核时
- **则** 可以批准或拒绝，填写审核意见
- **并且** 自动通知用户审核结果
