# 代码审查与测试规范

## 1. 代码 Review 流程

### 1.1 代码提交流程

```
开发者编写代码
    ↓
本地测试通过 (npm run lint, npm run typecheck)
    ↓
提交 Pull Request
    ↓
至少 1 名其他开发者 Review
    ↓
Review 通过后合并到主分支
    ↓
自动部署（如果 CI/CD 配置）
```

### 1.2 Review 时限

| 场景 | 响应时限 | 说明 |
|------|---------|------|
| 普通 PR | 24 小时内 | 正常工作日 |
| 紧急修复 | 4 小时内 | 需要快速响应 |
| 大型重构 | 48 小时内 | 需要更详细审查 |

### 1.3 Review 检查清单

#### 功能性检查
- [ ] 代码逻辑是否正确
- [ ] 是否处理了边界情况
- [ ] 错误处理是否完善
- [ ] 是否引入了新的 Bug

#### 代码质量检查
- [ ] 是否遵循命名规范
- [ ] 是否有重复代码（需要抽取）
- [ ] 注释是否清晰必要
- [ ] 函数是否过长（需要拆分）

#### 性能检查
- [ ] 是否有不必要的重复渲染（React）
- [ ] 是否有大循环或深递归
- [ ] 资源是否正确释放
- [ ] 懒加载是否合理使用

#### 安全检查
- [ ] 是否有 SQL 注入风险
- [ ] 是否有 XSS 风险
- [ ] 敏感信息是否硬编码
- [ ] 权限校验是否完善

#### 测试覆盖
- [ ] 关键逻辑是否有测试
- [ ] 测试用例是否充分
- [ ] 边界条件是否覆盖

## 3. 主动 Review 机制

### 3.1 发现问题主动介入

**规则**：任何 Agent 发现其他人的代码问题时，必须主动指出

**操作流程**：
1. 在相关代码处添加评论
2. 与代码作者讨论问题
3. 共同讨论解决方案
4. 代码作者进行重构
5. 确认修复完成

**示例**：
```
[@前端开发] 发现 order.ts 第 45 行存在潜在空指针风险：
- 问题描述：getOrderById 可能返回 undefined
- 建议：在调用前添加空值检查
- 讨论：同意此问题，我会添加防御性检查
```

### 3.2 代码完成后主动通知

**规则**：完成代码后主动通知相关 Agent 进行 Review

**通知内容**：
- 完成的功能模块
- 改动的主要文件
- 需要的 Reviewer
- 测试情况说明

## 4. 测试规范

### 4.1 测试分层

```
单元测试 (Unit Tests)
    ↓
集成测试 (Integration Tests)
    ↓
端到端测试 (E2E Tests)
```

### 4.2 测试覆盖率要求

| 类型 | 覆盖率要求 | 说明 |
|------|----------|------|
| 单元测试 | 70%+ | 核心业务逻辑 |
| 集成测试 | 50%+ | API 端点 |
| E2E 测试 | 核心流程 | 关键用户路径 |

### 4.3 前端测试规范

```typescript
// 测试文件命名
ComponentName.test.tsx

// 测试结构
describe('ComponentName', () => {
  it('should render correctly', () => {
    // 测试渲染
  });

  it('should handle user interaction', () => {
    // 测试交互
  });

  it('should display error state', () => {
    // 测试错误处理
  });
});
```

### 4.4 后端测试规范

```typescript
// 测试文件命名
module.service.spec.ts

// 测试结构
describe('UsersService', () => {
  describe('createUser', () => {
    it('should create a new user', async () => {
      // 测试创建用户
    });

    it('should throw error for duplicate email', async () => {
      // 测试错误情况
    });
  });
});
```

## 5. Bug 处理流程

### 5.1 Bug 报告模板

```markdown
## Bug 报告

**标题**: [简洁描述问题]
**优先级**: [P0/P1/P2/P3]
**复现步骤**:
1. [步骤1]
2. [步骤2]
3. [步骤3]

**预期行为**: [描述期望的结果]
**实际行为**: [描述实际的结果]
**环境**: [浏览器/操作系统等]
**截图**: [如有]

**分析**:
- 根本原因：[分析原因]
- 影响范围：[影响的功能]
```

### 5.2 Bug 修复流程

```
发现 Bug
    ↓
创建 Issue/标签 Bug
    ↓
指派给开发者
    ↓
修复代码
    ↓
添加测试用例
    ↓
Review 通过
    ↓
合并并部署
    ↓
验证修复
```

## 6. 质量门禁

### 6.1 合入标准

所有代码合入主分支前必须满足：

- [ ] ESLint 检查通过
- [ ] TypeScript 类型检查通过
- [ ] 所有单元测试通过
- [ ] 至少 1 人 Review 通过
- [ ] 测试覆盖率无明显下降（下降 > 5% 需说明）
- [ ] 无高/中优先级安全警告

### 6.2 CI/CD 自动化检查

```yaml
# .github/workflows/ci.yml
lint:
  - npm run lint
  - npm run typecheck

test:
  - npm run test
  - npm run test:coverage

build:
  - npm run build
```
