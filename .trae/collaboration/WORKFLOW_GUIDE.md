# MediMate 多Agent协作流程手册

## 概述

本文档定义了 MediMate 项目中 6 个固定角色 Agent 的协作流程规范，确保团队高效协同，输出统一、可运行的完整工程代码。

---

## 1. 固定角色定义

| 角色 | 职责定位 |
|------|----------|
| PM | 项目经理：需求拆解、规划、任务分配、交付汇总 |
| UI设计师 | 界面设计：页面结构、布局、配色、组件样式、交互逻辑 |
| 架构师 | 技术架构：技术栈选型、API规范、数据库设计 |
| 前端开发 | 前端实现：页面开发、样式实现、API联调 |
| 后端开发 | 后端实现：API编写、业务逻辑、数据库操作 |
| 测试工程师 | 质量保障：测试、代码审查、Bug追踪 |

---

## 2. 协作链路（固定执行顺序）

```
需求提出
    ↓
PM 规划
    ↓
UI 设计
    ↓
架构设计
    ↓
前端 & 后端 并行开发
    ↓
测试审查
    ↓
最终交付
```

---

## 3. 详细流程

### 3.1 需求阶段

**参与者**：PM 主导，所有 Agent 参与

**活动**：
1. PM 收集需求
2. 召开需求启动会议
3. PM 拆解任务并分配

**输出物**：
- 需求文档
- 任务清单
- 风险评估

---

### 3.2 设计阶段

**参与者**：UI设计师 + 架构师

**活动**：
1. UI设计师输出设计方案
2. 架构师设计技术方案
3. 召开方案评审会议
4. 输出设计文档

**输出物**：
- UI 设计稿
- API 规范文档
- 数据库模型

---

### 3.3 开发阶段

**参与者**：前端开发 + 后端开发（并行）

**活动**：
1. 前端开发接收设计稿和 API 规范
2. 后端开发实现 API
3. 前端开发实现页面
4. 前后端进行 API 联调
5. 代码完成后通知 Review

**输出物**：
- 前端代码
- 后端代码
- API 文档

---

### 3.4 审查阶段

**参与者**：测试工程师 + 所有开发者

**活动**：
1. 测试工程师开始测试
2. 开发者修复 Bug
3. 代码 Review 通过
4. 测试验证修复

**输出物**：
- 测试报告
- Bug 修复
- Review 通过

---

### 3.5 交付阶段

**参与者**：PM 主导，所有 Agent 参与

**活动**：
1. 功能验收
2. 文档同步
3. 版本发布
4. 项目汇总

**输出物**：
- 可运行的产品
- 更新后的文档
- 发布说明

---

## 4. 协作规则

### 4.1 核心规则

1. **先讨论后开发**
   - 每个功能必须经过团队讨论
   - 形成统一方案后再开始编码

2. **共同开发同一套代码**
   - 不划分固定边界
   - 看到不完善直接接手完善

3. **实时互相 Review**
   - 代码完成后必须被 Review
   - 发现问题直接讨论重构

4. **围绕同一目标推进**
   - 始终围绕当前功能点推进
   - 禁止各自为战

5. **完整交付**
   - 统一结构、统一规范
   - 输出完整可运行的工程

---

## 5. 禁止行为

| 禁止行为 | 说明 |
|---------|------|
| 割裂分工 | "你做UI、我做后端"式分工 |
| 独立片段 | 每人输出独立代码后简单拼接 |
| 不讨论开发 | 不讨论直接分头开发 |
| 视而不见 | 发现问题不指出 |
| 拒绝反馈 | 不接受修改意见 |

---

## 6. 任务分配规则

### 6.1 分配原则

- PM 根据角色职责分配
- 任务分配考虑个人技能
- 鼓励主动认领任务
- 阻塞任务优先处理

### 6.2 任务状态

| 状态 | 说明 |
|------|------|
| 待分配 | 未分配给任何人 |
| 已分配 | 分配给某人，待开始 |
| 进行中 | 正在开发 |
| 待Review | 代码完成，等待Review |
| 已完成 | Review通过，任务完成 |

---

## 7. 进度跟踪机制

### 7.1 每日站会

- 时间：每个工作日开始时
- 时长：15分钟以内
- 内容：汇报进度、计划、阻碍

### 7.2 周报

- 时间：每周五
- 内容：本周完成、下周计划、风险预警
- 发送：PM 汇总后发送至团队

### 7.3 里程碑检查

- 时间：每个功能节点
- 内容：检查功能完成情况
- 参与者：PM + 相关 Agent
