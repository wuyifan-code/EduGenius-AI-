# 角色职责与对接规范

## 1. 角色职责矩阵

| 角色 | 主要职责 | 直接对接 | 禁止行为 |
|------|----------|----------|----------|
| PM | 需求拆解、规划、任务分配、汇总交付 | 所有角色 | 不写具体代码 |
| UI设计师 | 界面方案、样式规范、交互流程 | 前端开发 | 不参与后端与架构 |
| 架构师 | 技术栈、API规范、数据模型 | 前端、后端 | 不参与UI设计 |
| 前端开发 | 页面开发、API联调 | UI、架构、测试 | - |
| 后端开发 | API实现、业务逻辑、数据库 | 架构、前端、测试 | - |
| 测试工程师 | 测试、审查、Bug追踪 | 所有开发角色 | - |

---

## 2. 详细职责说明

### 2.1 PM（项目经理）

**核心职责**：
- 需求收集与分析
- 任务拆解与分配
- 进度跟踪与协调
- 风险管理
- 最终交付汇总

**工作边界**：
- ❌ 不编写具体代码
- ❌ 不直接修改代码
- ✅ 协调资源解决阻塞
- ✅ 管理需求变更

**对接规范**：
- 与所有角色保持沟通
- 主持需求启动会议
- 汇总各方进度报告
- 协调冲突和资源

---

### 2.2 UI/交互设计师

**核心职责**：
- 页面结构设计
- 布局与配色方案
- 组件样式规范
- 交互流程设计
- 响应式适配方案

**工作边界**：
- ❌ 不参与后端 API 设计
- ❌ 不参与数据库设计
- ❌ 不编写后端代码
- ✅ 提供设计标注和说明
- ✅ 参与交互评审

**对接规范**：
- 只与前端开发对接
- 输出：设计稿、样式规范、交互说明
- 接收：前端实现反馈
- 评审前端实现是否符合设计

---

### 2.3 系统架构师

**核心职责**：
- 技术栈选型
- 前后端架构设计
- API 接口定义
- 数据库模型设计
- 系统安全性设计

**工作边界**：
- ❌ 不参与 UI 视觉设计
- ❌ 不编写前端样式代码
- ✅ 输出技术方案文档
- ✅ 提供 API 规范给前后端
- ✅ 评审架构可行性

**对接规范**：
- 对接前端：提供 API 规范
- 对接后端：提供数据模型和接口定义
- 评审前后端实现的架构一致性

---

### 2.4 前端开发工程师

**核心职责**：
- 页面组件开发
- 样式实现
- 交互逻辑编写
- API 对接与联调
- 前端性能优化

**工作边界**：
- ❌ 不编写后端 API 逻辑
- ❌ 不设计数据库
- ✅ 实现 UI 设计师的设计
- ✅ 调用架构师定义的 API
- ✅ 与后端协作调试

**对接规范**：
- 对接 UI：接收设计方案
- 对接架构：获取 API 规范
- 对接后端：联调 API 接口
- 对接测试：提供可测试的前端代码

---

### 2.5 后端开发工程师

**核心职责**：
- API 接口实现
- 业务逻辑编写
- 数据库操作
- 认证授权
- 数据验证

**工作边界**：
- ❌ 不编写前端组件
- ❌ 不设计 UI 样式
- ✅ 实现架构师设计的 API
- ✅ 提供可用接口给前端
- ✅ 编写接口文档

**对接规范**：
- 对接架构：遵循接口规范
- 对接前端：提供联调支持
- 对接测试：提供测试账号和数据

---

### 2.6 代码审查 & 测试工程师

**核心职责**：
- 单元测试编写
- 集成测试
- E2E 测试
- 代码审查
- Bug 追踪

**工作边界**：
- ✅ 可审查所有角色的代码
- ✅ 可直接指出代码问题
- ❌ 不直接修改他人代码（除非获得授权）
- ✅ 提供修改建议

**对接规范**：
- 对接所有开发角色
- 审查代码质量和规范
- 报告 Bug 并跟踪修复
- 验证修复完成

---

## 3. 角色间沟通流程

### 3.1 UI → 前端开发

```
UI设计师完成设计
    ↓
召开方案评审会议
    ↓
前端开发接收设计文档
    ↓
前端开发实现并反馈问题
    ↓
UI设计师验收
```

### 3.2 架构师 → 前端/后端

```
架构师完成架构设计
    ↓
召开技术评审会议
    ↓
输出 API 规范文档
    ↓
前端/后端按规范开发
    ↓
架构师评审实现一致性
```

### 3.3 前端 ↔ 后端

```
前端开发需要接口
    ↓
查看 API 规范文档
    ↓
有疑问时直接沟通
    ↓
进行接口联调
    ↓
发现问题互相反馈
```

### 3.4 测试 → 开发

```
开发完成功能
    ↓
测试工程师开始测试
    ↓
发现 Bug 提交 Issue
    ↓
开发修复 Bug
    ↓
测试验证修复
```

## 4. 冲突解决机制

### 4.1 冲突类型

| 类型 | 说明 | 解决方法 |
|------|------|----------|
| 技术冲突 | 技术方案争议 | 架构师最终决定 |
| 进度冲突 | 任务优先级争议 | PM 协调决定 |
| 质量冲突 | 代码质量标准争议 | 测试工程师+架构师决定 |
| 设计冲突 | UI 与实现争议 | UI 设计师+前端协商 |

### 4.2 解决流程

```
发生冲突
    ↓
双方自行协商（24小时内）
    ↓
无法解决 → 升级至相关角色负责人
    ↓
仍无法解决 → PM 裁决
    ↓
记录冲突原因和解决方案
```

### 4.3 升级路径

```
个人 → 双方协商
  ↓
无法解决
  ↓
PM 介入 → 裁决
```

## 5. 协作禁止行为

| 禁止行为 | 说明 | 后果 |
|---------|------|------|
| 割裂分工 | "你只做UI、我只做后端" | 破坏协作文化 |
| 独立片段 | 每人输出独立代码后拼接 | 代码不一致 |
| 不讨论开发 | 不经讨论直接分头开发 | 方向偏差 |
| 视而不见 | 发现问题不指出 | 遗留隐患 |
| 拒绝反馈 | 不接受他人修改意见 | 阻碍改进 |
