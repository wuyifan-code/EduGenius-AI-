# MediMate 多Agent团队协作规范

## 为什么
当前项目开发缺乏系统化的团队协作机制，各Agent之间缺乏明确的分工与沟通流程。需要建立一套完整的多Agent协作模式，确保6个固定角色的Agent能够高效协同工作，输出统一的、可运行的完整工程代码。

## 什么变化

### 组织架构定义
- **PM（项目经理）**：需求拆解、整体规划、任务分配、最终交付汇总
- **UI/交互设计师**：页面结构、布局、配色、组件样式、交互逻辑
- **系统架构师**：技术栈选型、架构设计、API规范、数据模型
- **前端开发工程师**：依据UI设计和API规范开发前端页面和交互
- **后端开发工程师**：依据架构设计实现API、业务逻辑、数据库操作
- **代码审查 & 测试工程师**：统一测试、代码审查、Bug修复、修改意见

### 协作链路（固定执行顺序）
1. 需求 → PM规划 → UI设计 → 架构设计 → 前端&后端并行开发 → 测试审查 → 最终交付
2. 所有Agent先讨论需求、技术方案、实现思路，形成统一方案后开始编码
3. 围绕同一个目标推进，禁止各写各的模块
4. 实时互相Review，发现问题直接讨论重构

### 禁止行为
- 禁止割裂分工（"你做UI、我做后端"式分工）
- 禁止每人输出独立片段最后简单拼接
- 禁止不讨论直接分头开发

## 影响
- 受影响的规范：所有后续功能开发规范
- 受影响的代码：整个项目工程
- 工作流程：引入多Agent协作评审机制

## 新增需求

### 需求：团队协作会议机制
所有功能开发前必须经过团队讨论

#### 场景：需求启动会议
- **当** 新功能需求提出时
- **则** PM主持会议，所有Agent参与讨论
- **并且** 讨论技术方案、实现思路、风险点
- **最终** 形成统一的实现方案文档

#### 场景：方案评审会议
- **当** UI设计师输出设计方案后
- **则** 架构师评审可行性
- **并且** 前后端评审接口设计
- **最终** 确认可执行的开发方案

### 需求：统一代码规范
所有Agent必须遵守统一的代码规范

#### 场景：代码风格统一
- **当** 任何Agent编写代码时
- **则** 必须遵循项目既有的代码风格和命名规范
- **并且** 使用项目统一的工具链

#### 场景：代码提交规范
- **当** 代码编写完成后
- **则** 必须经过至少一个其他Agent的Review
- **并且** 通过后才能合并到主分支

### 需求：实时协作与Review
代码必须经过实时Review机制

#### 场景：代码完成后主动Review
- **当** 一个Agent完成一段代码后
- **则** 主动通知其他Agent进行Review
- **并且** 根据反馈进行重构优化

#### 场景：发现问题主动介入
- **当** 某个Agent发现其他人的代码问题时
- **则** 直接指出、讨论、共同重构
- **禁止** 视而不见、等别人改

### 需求：统一交付标准
最终输出必须是完整可运行的工程

#### 场景：完整功能交付
- **当** 功能开发完成后
- **则** 必须包含完整的前端、后端、数据库
- **并且** 集成测试通过、可正常运行

#### 场景：文档同步更新
- **当** 代码发生变更时
- **则** 相关文档（API文档、设计文档）同步更新

## 修改需求

### 需求：现有开发流程调整
**修改内容**：引入多Agent协作评审机制

- 所有功能开发前必须经过团队讨论
- 代码完成后必须经过Review
- 发现问题必须主动重构

## 移除需求
- 无

## 实施细节

### 角色职责矩阵
| 角色 | 主要职责 | 对接角色 | 禁止行为 |
|------|----------|----------|----------|
| PM | 需求拆解、规划、任务分配、汇总交付 | 所有角色 | 不写具体代码 |
| UI设计师 | 界面方案、样式规范、交互流程 | 仅前端开发 | 不参与后端与架构 |
| 架构师 | 技术栈、API规范、数据模型 | 前端、后端 | 不参与UI设计 |
| 前端开发 | 页面开发、API联调 | UI、架构、测试 | - |
| 后端开发 | API实现、业务逻辑、数据库 | 架构、前端、测试 | - |
| 测试工程师 | 测试、审查、Bug追踪 | 所有开发角色 | - |

### 协作规则执行要点
1. **先讨论后开发**：每个功能必须经过团队讨论形成统一方案
2. **共同开发同一套代码**：不划分固定边界，看到不完善直接接手完善
3. **实时Review**：写完代码必须被Review，发现问题直接讨论重构
4. **围绕同一目标**：始终围绕当前功能点推进，禁止各自为战
5. **完整交付**：输出统一结构、统一规范、完整可运行的工程
